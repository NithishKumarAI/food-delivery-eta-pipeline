## Overview

Food delivery platforms need reliable time estimates to improve customer experience and operations planning. This project trains and deploys a regression model that predicts **delivery time in minutes** using signals such as courier profile, weather, traffic, order context, geographic distance, and temporal patterns.

**What this project demonstrates:**

| Area | Implementation |
|------|----------------|
| Data science | EDA, cleaning, leakage-aware train/test split |
| ML modeling | Baseline + tree-based model comparison |
| ML engineering | Reusable preprocessing and inference modules |
| Deployment | Docker image + Google Cloud Run + Streamlit UI |
| Software design | Modular `src/` layout, separation of training vs inference |

---

## Demo Disclaimer

This application is a **portfolio and educational demo**. It is not affiliated with any commercial delivery platform. Predictions are generated by a trained ML model on historical-style data and **should not be treated as operational guarantees** for real-world logistics systems.

## Key Highlights

- End-to-end pipeline from raw delivery records to deployed web app

## System Architecture

High-level request flow:

```text
User
  │
  ▼
Streamlit UI (app.py)
  │
  ▼
Inference Pipeline (src/preprocessing.py → src/predict.py)
  │
  ▼
CatBoost Model (models/catboost_model.cbm)
  │
  ▼
Predicted ETA (minutes)
```

```mermaid
flowchart LR
    A[User] --> B[Streamlit UI]
    B --> C[Preprocessing]
    C --> D[CatBoost Model]
    D --> E[ETA Prediction]
```

**Design notes:**

- The UI collects order and context inputs; preprocessing derives engineered features and enforces the exact training feature schema before prediction.
- The serialized CatBoost model is loaded once and reused across requests for efficient inference.
- The same inference code path powers both the Streamlit app and the optional REST API.

## Model Performance

Models were evaluated on a held-out test set (80/20 split, `random_state=42`) after feature engineering and removal of raw datetime fields used only for feature creation.

| Model | R² Score | Notes |
|-------|----------|--------|
| Linear Regression | 0.60 | Baseline; limited nonlinear capacity |
| Random Forest | 0.82 | Strong tree ensemble benchmark |
| XGBoost | 0.82 | Competitive gradient boosting |
| LightGBM | 0.83 | Fast training, strong accuracy |
| **CatBoost (production)** | **0.84** | **Best test performance; deployed model** |

*Metrics reflect offline evaluation on the project dataset. Production monitoring would require live feedback loops not included in this demo.*

---

## Why CatBoost for Production

CatBoost was chosen as the final deployed model because it offered the best balance of accuracy and deployment characteristics:

- **Highest test R²** among models trained in this project
- **Native categorical handling** — no one-hot expansion at inference, reducing schema complexity
- **Faster inference** than large forest ensembles in many deployment scenarios
- **Smaller deployment footprint** compared to bulky ensemble artifacts for comparable performance
- **Stable serialization** via `.cbm` for straightforward loading in Python services

## Feature Engineering

Engineered features mirror the training notebook and inference pipeline:

| Feature | Description |
|---------|-------------|
| `distance_km` | Geodesic distance between restaurant and delivery coordinates |
| `order_hour` | Hour of day extracted from order timestamp |
| `day_of_week` | Weekday name from order date |
| `is_weekend` | Boolean flag for Saturday/Sunday |
| `meal_time` | Meal period bucket (Breakfast, Lunch, Dinner, LateNight) from order hour |
| `preparation_time` | Minutes between order placed and order picked (or supplied at inference) |

Raw columns `Order_Date`, `Time_Orderd`, and `Time_Order_picked` are used only during preprocessing and are dropped before model input.

**Final model feature set (20 features):**  
`Delivery_person_Age`, `Delivery_person_Ratings`, `Restaurant_latitude`, `Restaurant_longitude`, `Delivery_location_latitude`, `Delivery_location_longitude`, `Weatherconditions`, `Road_traffic_density`, `Vehicle_condition`, `Type_of_order`, `Type_of_vehicle`, `multiple_deliveries`, `Festival`, `City`, `day_of_week`, `is_weekend`, `preparation_time`, `order_hour`, `distance_km`, `meal_time`

## Usage

### 1. Streamlit app (recommended for demo)

From the project root:

```bash
streamlit run app.py
```

Open the local URL shown in the terminal, adjust inputs, and click **Predict ETA**.

### 2. Python inference API

```python
import pandas as pd
from src.predict import predict_eta

order = pd.DataFrame([{
    "Delivery_person_Age": 29,
    "Delivery_person_Ratings": 4.7,
    "Restaurant_latitude": 12.9716,
    "Restaurant_longitude": 77.5946,
    "Delivery_location_latitude": 12.9352,
    "Delivery_location_longitude": 77.6245,
    "Weatherconditions": "Sunny",
    "Road_traffic_density": "High",
    "Vehicle_condition": 2,
    "Type_of_order": "Snack",
    "Type_of_vehicle": "Bike",
    "multiple_deliveries": 1,
    "Festival": "No",
    "City": "Urban",
    "Order_Date": "2024-05-26",
    "Time_Orderd": "18:45:00",
    "preparation_time": 20.0,  # or provide Time_Order_picked instead
}])

minutes = predict_eta(order)
print(f"Predicted delivery time: {minutes:.2f} minutes")
```

## Deployment

The application is packaged as a Docker container and deployed to **Google Cloud Run** for a scalable, publicly accessible demo.

### Docker (local)

```bash
docker build -t food-delivery-eta .
docker run -p 8080:8080 food-delivery-eta
```

Visit `http://localhost:8080`.

### Google Cloud Run (summary)

1. Build and push the image to **Google Artifact Registry** (or Container Registry).
2. Deploy a Cloud Run service using the image, port **8080**, and unauthenticated access if desired for a public demo.
3. Set memory/CPU appropriate for CatBoost + Streamlit cold starts (e.g., 1–2 GiB RAM for small traffic).

```bash
# Example flow (adjust project/region/image names)

  --image REGION-docker.pkg.dev/PROJECT_ID/REPO/food-delivery-eta \
  --platform managed \
  --region REGION \
  --allow-unauthenticated \
  --port 8080
```

## Engineering Learnings

- **Training and inference must share one preprocessing contract** — column order, dtypes, and engineered features must match exactly or production metrics will not hold.
- **Tree models need thoughtful benchmarking** — baselines (Linear Regression) establish a floor; ensembles and boosting methods capture nonlinear delivery dynamics.
- **Categorical-heavy logistics data favors CatBoost** — reduces encoding complexity and keeps inference pipelines maintainable.
- **Leakage prevention matters** — datetime fields used only for feature extraction should be dropped before model input.
- **ML engineering is more than training** — serialization, modular `src/` code, containerization, and a simple UI turn a notebook experiment into a demonstrable system.
- **Deployment constraints influence model choice** — accuracy, inference latency, and artifact size all matter for Cloud Run-style services.
